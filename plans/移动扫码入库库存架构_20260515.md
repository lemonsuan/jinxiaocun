## 推荐结论

改为“App-only + 单人单设备 + 本地优先 + 可选云端备份”的架构。MVP 不做 PC 端，所有核心操作都在安卓/iOS App 内完成：快递单号二维码扫码、商品清单拍照识别、人工确认入库、结算标记、历史入库查询、商品总量查询、商品出货查询。

推荐技术路线是 Flutter 跨平台 App + Android/iOS 原生扫码桥接 + PaddleOCR Mobile/PP-Structure 端侧表格识别 + 自定义后处理规则 + 本地 SQLite 数据库。库存统计类 App 的核心价值是离线可用和账目准确，因此本地库存流水是权威数据源；联网时只做云端备份，不参与库存判断。

## 系统分层

### App 入库与查询

* 创建入库单。
* 调用 Android/iOS 原生能力扫描快递单号二维码，识别失败时允许手动输入。
* 拍摄商品清单，使用 PaddleOCR Mobile + PP-Structure 在本机识别表格、商品名称和数量。
* 展示 OCR 商品明细草稿：产品编号、产品名称、数量、进价、卖价。
* 用户人工确认后提交入库。
* 对每一单自由标记已结算/未结算。
* 查询历史入库记录、商品总量、商品出货记录。
* 创建出货记录并扣减库存。
* 离线保留全部草稿、图片、OCR 结果、库存流水和查询数据。

### 暂不做 PC 端

* PC 管理台、后台审核和复杂报表不进入 MVP。
* 如果后续需要，可以复用同一套云端 API 增加 Web 管理端。

### App 本地服务

* Flutter App Shell：共享页面、路由、状态管理和业务用例。
* Local DB Service：SQLite 本地数据库、事务、索引、迁移。
* Scanner Service：快递单号二维码扫码，失败时手动录入。
* OCR Service：通过 PaddleOCR Mobile + PP-Structure 本机识别商品清单图片，输出表格结构化草稿。
* Ocr Post-Processor：清洗 OCR 文本、修正数量列、合并换行商品名、过滤页眉页脚、按产品编号/名称/数量映射成入库明细草稿。
* Native Bridge：Flutter MethodChannel 封装 Android/iOS 扫码与 PaddleOCR/PP-Structure 调用。
* Inbound Service：入库单、入库明细、结算标记。
* Inventory Service：库存流水、库存汇总、负库存校验。
* Outbound Service：出货记录、出货明细、确认扣库。
* Product Service：商品编号、名称、别名匹配。
* Sync Service：联网后可选同步云端、备份图片和业务数据。

### 可选云端服务

* API 服务：数据备份和恢复。
* 对象存储：联网后备份快递面单、商品清单原图和压缩图。
* 关系型数据库：保存云端副本，便于后续 PC 端或多设备使用；MVP 不把它作为权威库存。
* 日志与监控：同步失败、识别失败、库存事务失败必须可追踪。

## 核心数据模型

* `products`：商品主档，产品编号、名称、规格、默认价格。
* `product_aliases`：OCR 名称/历史名称/供应商名称到商品的映射。
* `warehouses`：仓库；MVP 只启用一个默认仓库。
* `inbound_receipts`：入库单，含快递单号、商品清单图片、结算状态、OCR 状态。
* `inbound_items`：入库明细，商品编号、名称、数量、进价、卖价。
* `outbound_orders`：出货记录/出库单。
* `outbound_items`：出库明细。
* `stock_ledger`：库存流水，唯一可信账本。
* `warehouse_stock`：本地库存汇总表，由流水聚合而来。
* `attachments`：本地图片和文件元数据。
* `ocr_results`：OCR 结果 JSON、置信度、失败原因。
* `sync_queue`：待备份操作队列，支持联网后重试。

## 关键流程

### App 离线入库流程

1. 用户打开 App 并创建入库单。
2. 扫描快递单号二维码，失败时手动输入。
3. 拍商品清单，图片保存到本地 App 沙盒。
4. 本机 PaddleOCR + PP-Structure 识别商品清单表格，状态从 `pending` 到 `needs_review`。
5. 自定义后处理规则将表格结果转成入库明细草稿。
6. 用户在手机上核对 OCR 明细，必要时修正产品编号、名称和数量。
7. 用户确认入库，App 在本地事务内写入入库单、库存流水、库存汇总。
8. 用户按需要标记该单“已结算/未结算”。
9. 系统标记快递单号已入库，防止重复提交。
10. 联网后 App 将入库单、图片和库存流水加入备份队列上传。

### App 出货流程

1. 用户创建出货记录/出库单。
2. 搜索或选择商品，填写数量。
3. App 检查本地可用库存。
4. 确认出货后写 `stock_ledger` 扣减流水。
5. 更新本地库存汇总并生成可追溯出货记录。

### App 查询流程

* 历史入库查询：按日期、快递单号、商品编号、结算状态筛选。
* 商品总量查询：按商品编号/名称聚合当前库存。
* 商品出货查询：按日期、商品编号、出货单查询出货流水。

## 风险控制

* 快递单号唯一索引，防重复入库。
* 入库/出库确认必须使用本地数据库事务。
* OCR 只生成草稿，人工确认后才影响库存。
* 库存变化只通过 `stock_ledger`，禁止直接改库存数量。
* 默认不允许负库存；出货数量超过当前库存时阻止确认。
* 原图、OCR 原始结果、人工修正结果都要保留。
* 同步/备份队列不得改变本地已确认账本，只负责上传云端副本。

## MVP 建议

1. App：扫码/手输快递单号、拍照、PaddleOCR + PP-Structure 表格识别、后处理草稿确认、提交入库。
2. App：入库单结算标记、历史入库查询、商品总量查询、商品出货查询。
3. App：创建出货记录，确认后扣减库存。
4. 数据：本地 SQLite、单仓库、价格可空、快递单号防重复、库存流水。
5. 备份：先作为可选模块，支持联网后云端备份。

## 当前限制

* iOS 保留 PaddleOCR Mobile + PP-Structure SLANet。Android OCR 已改为 PP-OCRv5 LiteRT/TFLite，在独立 `:ocr` 服务进程内加载 `libinventory_ocr.so`、`libLiteRt.so`、`libLiteRtOpenClAccelerator.so` 和 TFLite 模型。
* Android 表格结构不依赖 SLANet；native 层按 OCR 文本框坐标分组为行，Dart 后处理继续负责产品编号、中文商品名称和数量解析，保证仍可进入人工可编辑草稿。
* Android release APK 的真实 OCR 运行时仅支持 `arm64-v8a`；`armeabi-v7a` 和 `x86_64` 仅打包 no-op JNI stub 以避免 Flutter 默认多 ABI 构建失败。
* Android release APK 使用 Flutter 模板 release 签名配置，适合本机安装测试，不适合商店发布。
* iOS 已通过无签名编译与 archive 验证；没有 Apple Developer 签名证书时无法导出可直接安装真机的 IPA。
